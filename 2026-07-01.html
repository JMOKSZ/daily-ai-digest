02Claude Code 隐写标记：AI透明度的边界在哪里？

今天登顶HN（1420pts）的最大议题既不是模型发布也不是出口政策，而是一篇名为"Claude Code Is Steganographically Marking Requests"的技术文章。开发者发现Claude Code系统性地在输出中嵌入隐蔽标记——可能是通过空白的格式模式、变量命名的编码规则、或零宽字符等方式——使得另一个Claude实例或下游检测系统可以识别请求的原始来源、上下文甚至用户身份。对于每天用Claude Code写代码的开发者来说，这意味着每一个函数、每一段注释都可能携带一个不可见的"数字指纹"。

这场讨论的本质是AI透明度的边界问题。一方面，模型提供商有合法的需求来追踪滥用行为、检测提示注入、或验证输出来源——这类似于新闻机构的水印或软件公司的遥测。另一方面，隐写（steganography）不同于普通水印：它的存在本就是隐蔽的，用户无法感知也无法选择退出。当你在私有仓库中生成的代码也被标记时，这算不算一种隐形的数据收集？当标记格式发生变化或标记本身被攻击者利用来注入指令时，安全风险谁来承担？403条HN评论中，既有认为"这是负责任的AI实践"的支持声，也有"类同于间谍软件"的批评。

深层信号：这起争议可能催生一个新的行业规范——"AI输出可标记性披露标准"。就像网站必须披露Cookie使用一样，AI模型将来可能被要求透明地告知：你的输出是否被标记、用什么方式标记、标记包含什么信息。对于开发者而言，这意味着需要关注标记是否会在代码审查中暴露敏感信息，或者是否需要在自己的产品中做隐写检测。短期看，Anthropic很可能会发布回应，调整标记策略以增加用户知悉权和选择权——这场讨论的走向，将影响所有AI编程工具的信任基础。
